Best answer: **combine cheap wake suppression + cascade judging + extract-and-evaluate review + explicit validation-provenance checks**. That is the most plausible path to **lower price and higher detection quality**.

Legend: **price ↓** means cheaper, **quality ↑** means better detection, **↔** means likely neutral, **?** means must be validated on your own supervisor traces.

The most useful implementation shape would be:

`event → deterministic gate → cheap classifier → evidence extractor → strong judge only on risky/ambiguous cases → panel only on unresolved completion reviews`

For your failure modes, the highest-value detectors are probably:

`test edited by coder + trusted as validation`, `validation command claimed but not executed`, `tests run before final code changes`, `benchmark/verifier/harness modified`, `same failing command repeated`, `large unrelated diff`, `final “done” without behavioral evidence`, `mocking/patching external dependency to satisfy tests`, and `new tests only cover new implementation rather than task behavior`.

Cost-only techniques, so you do not confuse them with quality wins:

| Technique                                      |         Price effect |                                                                                                                                                                                                     Quality effect |
| ---------------------------------------------- | -------------------: | -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------: |
| **Prompt caching / prefix caching**            |                **↓** | **↔**. Same prompt, cheaper/faster repeated prefix. OpenAI docs say prompt caching can cut latency and input-token cost for repeated prefixes and does not change output generation. ([developers.openai.com][12]) |
| **Response caching / decision memoization**    |                **↓** |                                                                                                                              **↔/?**. Safe only when task, diff, command output, and policy version are identical. |
| **Batch / flex processing**                    |                **↓** |                                                                                 **↔**. Useful for offline evals and delayed completion reviews, not synchronous runtime supervision. ([developers.openai.com][13]) |
| **Speculative decoding**                       |        **latency ↓** |                                                                                                                                    **↔**. Usually preserves model distribution; mostly speed, not better judgment. |
| **Quantization**                               |     **infra cost ↓** |                                                                                                                      **↔/↓**. Can hurt small-margin detection; validate on false-done and masked-validation cases. |
| **KV-cache compression**                       | **memory/latency ↓** |                                                                                                                                                **↔/↓**. Useful for serving economics, not a detection improvement. |
| **Smaller model substitution without cascade** |                **↓** |                                                                                                                                               **↓/?**. Dangerous unless measured against your false-accept target. |

I would not bet on “cheaper and better” claims from OpenRouter Fusion-style panels for this use case. Panels can improve hard calls, but per-call fanout is inherently cost-up unless routed only to rare ambiguous cases. Vendor benchmark claims are not enough evidence for runtime supervision.
